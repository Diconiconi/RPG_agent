# 沉浸式自由输入对话 UI 设计

## 背景

当前 AI 对话功能主要通过 mod 自建的大聊天窗口完成。这个窗口适合配置 API、查看历史和调试，但正常游玩时会遮挡较多游戏画面，代入感弱于原游戏的 NPC 对话表现。

用户希望保留大聊天窗口，但让常规交谈流程更贴近原游戏：从 NPC 交互菜单进入“交谈”，在交谈选项里选择“自由输入”，输入玩家文本后隐藏输入框，再用原游戏底部对话框和 NPC 立绘显示 AI 回复。

## 目标

- 大聊天窗口继续存在，用于 API 配置、历史记录和后续管理功能。
- 不按原快捷键 `B` 时，大聊天窗口不因点击“交谈”自动弹出。
- 在原游戏“交谈”选项页新增“自由输入”入口。
- 点击“自由输入”后，在屏幕底部显示轻量输入框。
- 玩家提交消息后，输入框消失，后台调用现有 AI 对话流程。
- NPC 回复优先使用原游戏 `SayDialog` 展示，表现为底部文本框、NPC 名字、回复文本和 NPC 立绘。
- 现有好感度修改、战斗触发、对话历史记录继续生效。

## 非目标

- 第一版不重做大聊天窗口 UI。
- 第一版不新增新的好感度规则、战斗规则或任务分支规则。
- 第一版不把所有原游戏交谈选项替换为 AI 选项。
- 第一版不删除当前聊天历史、配置面板或调试能力。

## 推荐方案

采用“沉浸式交谈为主，大窗口工具化保留”的方案。

正常 NPC 交谈不再自动打开大聊天窗口。AI 对话入口被放进原游戏交谈选项中，名字为“自由输入”。玩家使用该入口时，只显示一个短输入框；提交后输入框隐藏，AI 回复通过原游戏 `SayDialog` 播放。

大聊天窗口仍由快捷键或配置入口打开，定位调整为工具面板。后续可以逐步删减其中的聊天输入区，只保留历史记录、角色卡、上下文、API 配置等功能。

## 交互流程

1. 玩家点击地图或场景中的 NPC。
2. 游戏显示 NPC 交互面板。
3. 玩家点击“交谈”。
4. 游戏显示原交谈选项列表。
5. Mod 在选项列表中加入“自由输入”。
6. 玩家点击“自由输入”。
7. 交谈选项列表关闭，屏幕底部显示轻量输入框。
8. 玩家输入文本并提交。
9. 输入框立即隐藏，避免遮挡游戏画面。
10. Mod 调用现有 AI 请求流程，保留上下文和历史记录。
11. AI 返回后，Mod 解析可见文本和隐藏效果指令。
12. Mod 使用 `SayDialog` 展示 NPC 回复。
13. 回复展示完成后，Mod 执行并反馈好感度变化或战斗触发等现有效果。

## 代码设计

### 入口控制

当前 `AIChatManager.StartChatWithNpc` 会在检测到 NPC 交谈后调用 `UIManager.ShowChatPanel`。新设计需要拆分这个行为：

- 点击 NPC 交谈时，只初始化当前 NPC、AI 上下文和历史文件。
- 不自动显示大聊天窗口。
- 大聊天窗口只在快捷键 `B` 或明确工具入口触发时显示。

### 自由输入选项

优先寻找原游戏交谈选项页的创建方法，在原列表中直接插入“自由输入”。这是视觉一致性最好的方式。

如果原列表注入不稳定，则使用备用方案：在点击“交谈”后，用 `MenuDialog` 显示一个兼容原风格的选项菜单，并加入“自由输入”。备用方案不作为最终体验首选，但可以降低第一版落地风险。

### 轻量输入框

新增一个独立的输入框组件，职责只包括：

- 显示在屏幕底部。
- 接收玩家输入。
- 支持提交和取消。
- 提交后隐藏自身。
- 调用统一的自由输入发送方法。

它不负责显示长历史，不负责 API 配置，不替代大聊天窗口。

### AI 回复与原生对话框

复用当前 `NPCDialog` 和 `AIChatManager.GetAIResponseAsync` 的请求能力，但需要让发送流程支持“不要追加到大聊天窗口，只返回 NPC 可见回复并继续应用效果”。

NPC 回复展示优先使用现有项目中已经出现过的 `SayDialog` 模式：

- `SayDialog.GetSayDialog()`
- `SetCharacter(null, npcId)`
- `SetCharacterImage(null, npcId)`
- `Say(replyText, ...)`

如果某些 NPC 立绘无法显示，则降级为只显示底部文本框，不中断对话。

## 数据流

玩家输入文本进入自由输入发送方法后：

1. 写入当前 NPC 的对话历史。
2. 调用 AI 接口获得回复。
3. 解析隐藏 `<dialog_effect>` 指令。
4. 将 NPC 可见回复写入历史。
5. 使用 `SayDialog` 展示可见回复。
6. 结算现有对话效果，包括好感度变化和战斗触发。

这个数据流应尽量复用当前大聊天窗口的发送逻辑，避免出现“大窗口聊天”和“沉浸式聊天”两套效果判断。

## 异常处理

- API 配置未完成：显示大聊天窗口或配置面板，引导用户配置 API。
- API 请求失败：用原生对话框或简短系统提示显示错误，不吞掉异常日志。
- 当前 NPC 为空：不显示自由输入入口，或者点击后给出简短提示。
- AI 回复为空：显示“对方沉默不语。”一类短文本，并记录日志。
- `SayDialog` 获取失败：回退到大聊天窗口追加回复，保证功能可用。
- 触发战斗时：优先先完成 NPC 回复展示，再执行现有战斗触发流程。

## 验证方式

- 点击 NPC 后，点击“交谈”，大聊天窗口不会自动出现。
- 原交谈选项中能看到“自由输入”。
- 点击“自由输入”后，底部输入框出现。
- 输入文本并提交后，输入框消失。
- AI 回复显示在原游戏底部对话框内。
- NPC 立绘能正常显示；无法显示时文本框仍可显示回复。
- 对话历史仍能在大聊天窗口中查看。
- 正向好感度仍遵守聊天上限 `60`。
- 严重冲突仍能触发当前已有战斗逻辑。
- 快捷键 `B` 仍能打开大聊天窗口进行 API 配置。

## 风险

- 原游戏交谈选项页的注入点可能不稳定，需要先定位具体 UI 创建方法。
- `SayDialog` 对不同 NPC 的立绘 ID 支持可能不完全一致。
- 输入框和游戏原 UI 的焦点可能冲突，需要测试回车、鼠标点击和取消行为。
- 如果玩家连续快速触发自由输入，需要防止重复请求和重复播放。

## 第一版完成标准

- 正常游玩时，AI 对话可以不依赖大聊天窗口完成一轮自由输入和 NPC 回复。
- 大聊天窗口不再因普通“交谈”自动弹出。
- 现有 AI 上下文、历史记录、好感度效果和战斗触发没有回退。
- 失败时有可见提示或稳定回退，不会卡住游戏界面。
