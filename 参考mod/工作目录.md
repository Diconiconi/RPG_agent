# RPG_agent 工作目录

更新日期：2026-06-10，Asia/Shanghai

## 当前方向
- 用户决定不再另起一个自有 mod 工程，而是直接基于 `参考mod` 改善现有 AI 对话 mod。
- 工作区应以 `参考mod` 为主体；后续开发优先研究并改造 `参考mod/plugins/MCS_AIChatMod.dll` 及其周边配置。
- 不要修改真实游戏安装目录。`F:\SteamLibrary\steamapps\common\觅长生` 默认只读。
- 不要提交 API key、认证文件、游戏存档、运行日志、真实游戏文件或复制来的二进制依赖。

## 已确认的设计决策
- 第一版目标：通过 AI 对话真实修改游戏内 NPC 好感度/关系，不再写入其他关系数值通道。
- 关系变化方式：AI 只输出“建议效果”，Mod 负责校验和执行。
- 玩家反馈：游戏内显示简短提示，例如“对方态度略有缓和：好感度 +2”。
- 单轮变化范围：按事件分档校验；通过聊天获得的正向好感度最高到 `60`，达到或超过 `60` 后不再通过聊天增加好感度。
- 负向对话效果不受 `60` 上限限制，辱骂、威胁、背叛等仍可降低好感度。
- 战斗触发：暂不进入第一版，只预留怒气/敌意记忆字段；等好感度写回稳定后再做第二阶段。

## 推荐机制草案
每轮玩家发言后，AI 正常生成 NPC 回复，同时附带一段隐藏结构化效果建议：

```json
{
  "favor_delta": 2,
  "reason": "玩家语气尊重，并主动表达善意",
  "memory_summary": "玩家曾以礼相待，使该 NPC 态度稍有缓和"
}
```

Mod 执行前必须校验：
- `favor_delta` 按事件分档强制夹到安全范围。
- 正向 `favor_delta` 执行前读取 `NPCEx.GetFavor(npcId)`；当前好感度达到或超过 `60` 时归零，超过 `60` 的部分裁剪掉。
- 无明确情绪或关系变化时归零。
- 同一轮只允许一次好感度变化。
- 只有校验通过后才调用 `NPCEx.AddFavor(npcId, delta)`。
- 成功后显示简短游戏提示。
- 同时记录 NPC 长期记忆，影响后续口吻和关系阶段。

## 现有参考结构
- `plugins/prompt.txt`：原 AI 对话总 prompt。
- `plugins/MCS_AIChatMod.dll`：当前参考 mod 主 DLL。
- `plugins/MCS_AIChatMod.pdb`：调试符号，仅作本地参考，不建议提交。
- `plugins/RPGSkillKit/`：之前整理出的 AI NPC skill/prompt 架构草案。
- `plugins/RPGSkillKit/state/npc_state.schema.json`：已有 `favor`、`relationship_stage` 等状态字段。
- `plugins/RPGSkillKit/memory/npc_memory.schema.json`：已有 `dialog`、`combat`、`relationship` 等记忆类型。
- `plugins/RPGSkillKit/config/skill-router.json`：已有按 NPC、势力、关系阶段加载 skill 的路由想法。

## 已观察到的潜在接入点
之前通过反射临时查看过游戏与参考 mod 类型。临时输出文件已按清理要求删除，关键结论记录如下：

- 游戏侧已采用的好感度接口：
  - `NPCEx.AddFavor`
  - `NPCEx.GetFavor`
  - `OfficialNpcFavorChangedPatch` 已监听 `NPCEx.AddFavor`，可通知任务系统好感变化。
  - `CmdWeiXieNPC`
  - `NewNpcFightManager`
  - `NpCFight`
  - `JSONClass.NpcHaoGanDuData`

  - `NPCDialog.SaveDialogEntry`
  - `MCS_AIChatMod.AIChatManager.GetAIResponseAsync`
  - `MCS_AIChatMod.UIManager.ShowChatPanel`

这些名字只能说明“方向上可能有入口”。下一步必须用反编译/符号查看确认调用签名、执行时机和是否会影响存档。

## 推荐下一步
1. 反编译或查看 `MCS_AIChatMod.dll` 的可读源码结构，优先找：
   - 玩家发送消息入口。
   - AI 回复解析入口。
   - NPC 当前 ID/名称获取方式。
   - 游戏提示/通知 UI 工具。
   - 现有好感变化 patch 的实现方式。
2. 建立 `DialogEffect` 数据结构：
   - `favor_delta`
   - `reason`
   - `memory_summary`
   - 后续预留 `anger_delta`、`trigger_candidate`
3. 修改 prompt，让模型以稳定格式返回“回复正文 + 隐藏效果 JSON”。
4. 增加解析和安全校验层。
5. 调用 `NPCEx.GetFavor` 和 `NPCEx.AddFavor` 执行真实好感度修改，并保持聊天正向收益 `60` 封顶。
6. 成功后写入 NPC 记忆，并显示简短提示。
7. 在游戏内用小范围 NPC 测试正向、负向、中性三类对话。

## 风险和边界
- 不应让 AI 直接执行游戏命令；AI 只能建议，Mod 必须裁决。
- 单轮关系变化要小，避免一句话破坏长期养成节奏。
- 战斗触发属于第二阶段，必须在好感度系统稳定后再接。
- 如果未来需要提交 GitHub，二进制 DLL/PDB/XML 依赖应继续被 `.gitignore` 或选择性提交策略保护。
